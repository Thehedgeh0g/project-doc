**Дата:** 10 декабря 2025 г.
**Статус:** Принято

# Контекст
Проект будет развиваться постепенно, добавляя новые функции:
- бронирование,
- уведомления,
- штрафы,
- отчёты,
- интеграции.

Команда небольшая, но роли разделены:
- backend-разработчики,
- frontend-разработчики,
- архитектор,
- тестировщик.

Необходимо выбрать технологический процесс, который обеспечит:
- предсказуемость поставок,
- прозрачность,
- качество,
- минимизацию рисков.

# Рассмотренные варианты
- **Классическая Waterfall** - Последовательная модель разработки.
- **Scrum** - Гибкая методология с двухнедельными спринтами.
- **Kanban** - Потоковая модель, ориентированная на непрерывную поставку.
- **Scrumban** - Гибрид Scrum и Kanban.

# Решение
Выбран **Scrum** с двухнедельными спринтами.

# Обоснование
- **Прогнозируемость** — каждые 2 недели команда выдаёт инкремент.
- **Управляемость** — ежедневные стендапы помогают отслеживать состояние задач.
- **Прозрачность** — каждая роль понимает свои задачи на спринт.
- **Подходит для проектной работы** — библиотечная система имеет большие фичи, которые разбиваются на спринты.
- **Удобство для тестировщиков** — тестирование переносится в конец спринта, есть стабильная итерация.

Scrum лучше подходит, чем Kanban, поскольку проект требует структурированных этапов и регулярных релизов.

# Недостатки

- Требует дисциплины со стороны команды.
- Планирование спринтов может занимать время.
- Менее гибок, чем Kanban, при экстренных задачах.

# Последствия

- Разработка идёт двухнедельными спринтами.
- Каждая задача должна быть оценена (story points).
- Проводятся стандартные Scrum-мероприятия:
  * планирование,
  * стендапы,
  * демо,
  * ретроспектива.
- Backlog должен поддерживаться аналитиком/архитектором
