# Контекст

Система Учёта Библиотеки должна обеспечивать регистрацию читателей, поиск книг, бронирование, выдачу и возврат, начисление штрафов, аналитику и уведомления.
Система будет использоваться одновременно библиотекарями, читателями и администраторами, поэтому важны стабильность, простота развёртывания и возможность дальнейшего расширения.

Необходимо выбрать архитектурную форму, которая обеспечит баланс между простотой разработки и возможностью масштабирования системы при росте нагрузки.

## Рассмотренные варианты

### Модульный монолит
Одно приложение, в котором функциональные области разделены на модули.

### Микросервисы
Каждая функциональная зона (читатели, каталог, выдача, штрафы, уведомления) — отдельный сервис.

### Серверлесс архитектура
Функции выполняются по событиям (например, выдача книги → триггер).

### Решение

Выбран модульный монолит.

### Обоснование

- Простота разработки и сопровождения — одна кодовая база, единый деплой.
- Минимальные накладные расходы — нет межсервисной коммуникации, меньше инфраструктуры.
- Достаточно для текущего масштаба — библиотека — не высоконагруженный сервис.
- Возможность роста — модули можно выделять в микросервисы при расширении.

### Недостатки

- Может ухудшиться масштабируемость при большом росте функционала.
- Требует чёткой внутренней модульности, чтобы избежать хаоса в коде.

### Последствия

- Необходим строгий контроль разделения модулей.
- При расширении функционала возможно выделение подсистем (например, каталога) в отдельные сервисы.
