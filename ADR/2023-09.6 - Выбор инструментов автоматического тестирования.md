# Контекст

Для проекта «Система Учёта Библиотеки» необходимо обеспечить стабильность и предсказуемость разработки.
Поскольку приложение будет включать:

- серверную часть (Go),
- клиентскую часть (React),
- взаимодействие с AMQP-брокером,
- работу с базой MySQL,

Нам требуется выбрать набор инструментов автоматического тестирования для:
- unit-тестов,
- интеграционных тестов,
- тестирования API,
- тестирования UI.

Инструменты должны быть надёжными, широко поддерживаемыми и подходящими под выбранный стек.

# Рассмотренные варианты
## Для backend (Go):

- Go testing package — встроенная библиотека тестирования Go.
- Testify — расширение для более удобных ассертов и моков.
- Ginkgo / Gomega — BDD-фреймворк для тестов.
- Postman/Newman — для тестирования API.

## Для frontend (React):

- Jest — де-факто стандарт для unit-тестов в JS.
- React Testing Library — тестирование UI на уровне поведения.
- Cypress — полноценное E2E-тестирование.
- Playwright — современная альтернатива для E2E.

## Решение

### Выбран набор:

### Backend (Go)
- Go testing package
- Testify
- Postman/Newman (для API)

### Frontend (React)
- Jest
- React Testing Library
- Cypress (E2E)

## Обоснование
### Backend
- Go testing package — встроенный, надёжный, простейший способ писать unit-тесты.
- Testify — делает тесты компактнее и читаемее, упрощает мокирование.
- Newman/Postman — удобный способ тестировать REST API и запускать коллекции в CI.

### Frontend
- Jest — быстрый, надёжный, идеально подходит для React.
- React Testing Library — фокус на тестировании поведения, а не реализации.
- Cypress — лучший инструмент E2E для сложных пользовательских сценариев (поиск книг, выдача, бронирование).

### Недостатки
- Cypress требует более тяжёлой инфраструктуры, чем Jest.
- Тестирование AMQP придётся реализовать вручную через мок-брокер или тестовый контейнер.

# Последствия
- Все тесты должны использовать выбранные инструменты.
- Тестирование становится обязательным этапом CI.
- Команда должна освоить Cypress.
